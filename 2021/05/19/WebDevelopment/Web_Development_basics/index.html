    기초 지식

    서버와 클라이언트

    서버(Server) : 서비스를 제공하는 쪽

    클라이언트(Client) : 서비스를 제공받는 쪽

    예시 : 페이스북 접속

    1. 브라우저를 킨다
      • 브라우저는 웹 클라이언트 (웹 페이지를 제공받으려는 쪽)
    2. 주소창에 www.facebook.com 입력
      • 웹 서버에 내용을 달라고 요청(Request)
    3. 사이트에 접속 (페이지가 뜬다)
      • 웹 서버가 응답(Response)함
      • HTML, CSS, JavaScript 등

    HTTP (HyperText Transfer Protocol)


    출처 : https://www3.ntu.edu.sg/home/ehchua/programming/webprogramming/HTTP_Basics.html

    HTTP 상태코드

    상태코드 설명
    1xx 조건부응답
    2xx 응답성공
    3xx 리다이렉션
    4xx 요청오류(ex 404 Not Found) (​클라이언트 오류)
    5xx 서버오류

    참고 : 서버는 응답함 json 형식으로

    5 가지 메서드

    HTTP Verbs (protocol, method) 의미
    GET 서버에게 정보를 보내달라고 요청
    브라우저에 google.com을 입력하면 google서버에 get 신호가 요청(request)되고 서버에서는 이에 응답(respond)하여 페이지를 클라이언트로 보내게 됨
    POST 클라이언트에서 서버에게 정보를 보내고 받아달라고 요청
    (form 작성, 로그인에 사용)
    PUT 클라이언트에서 서버에게 정보를 보내고 기존의 정보를 업데이트 해달라고 요청 (포스트 업데이트/수정)
    PATCH PUT과 차이 뭔지모르겠음
    DELETE 서버에 정보의 삭제를 요청 (포스트, 댓글을 삭제시 사용)

    네이티브 앱 vs 웹 앱 vs 하이브리드 앱

    • native app : 어플리케이션을 만들 때, OS에서 제공하는 기능을 사용

      • 원래 쓰던 방법, OS 바로 위에서 돌아가는 브라우저
      • 특징 : Android, ios 처럼 버전이 따로 있음 (개발 두번 필요)
      • 장점 : 스마트폰의 자원을 잘 활용 가능
      • 단점 : 급한 수정사항이나 업데이트 시 불편

    • 웹 앱 : 특정 주소로 들어가지는 웹인 앱

      • 장점 : 그때 그때 변경하기 좋음
      • 단점 : 게임이나 영상편집 등 고성능의 앱은 못 만듦

    • Hybrid app : 특정 부분에 웹을 올림(브라우저 사용)

      • Web 부분 변경 : Sever에 있는 원본을, 웹 클라이언트 개발자가 변경 (서버 개발자가 바꾸는거 아님)

        • 웹 클라이언트 개발자들이 서버까지 올림 (보통 그정도 함)

      • App 부분 변경 : 어플리케이션 개발자가 변경하고, 업데이트 시켜야 함

    적응형 웹 vs 반응형 웹

    • 적응형(adoptive) 웹 : 페이지 크기(디바이스)에 따라 페이지 주소가 우회됨 (모바일용 사이트 주소 따로 있음)
      • 사용 : 사진 크기 조절 등 레이아웃에 있어 뚜렷한 변화가 있을 때 차라리 따로 만드는게 좋음
    • 반응형(responsive) 웹 : 창 크기, 확대/축소 등에 맞춰 반응하여 레이아웃이 바뀌는 웹 (주소이동은 없음)
      • 사용 : 간단한 정보들, 데이터 배치가 바뀌어도 상관없을 때 (홈페이지 하나로 모든걸 커버하는거라 편함)

    쿠키 vs 세션 vs 캐시

    • 쿠키 : 사이트를 만든 개발자가 특정정보를 사용자의 브라우저(컴퓨터)에 저장되도록 프로그래밍 해놓은 것
      • 브라우저에 저장
      • 조작되거나 공개되어도 크게 상관 없는 정보 - 최근 검색 기록, 로그인 안했을 때 장바구니 정보 등 (보안, 결제는 절대 안됨)
    • 세션 : 사용자가 사용했을 때, 서버 컴퓨터에 필요한 내용을 저장
      • 서버에 저장
      • 서버의 자원을 사용하기 때문에 정말 중요한 것만 저장함 (보안, 결제 등)
    • 캐시 : 페이지가 변경 될 때 마다 다시 로그인할 수 없으니까, 캐시를 사용

      • 세션 키라 불림

      • 서버에게 이 세션의 주인임을 알려줌

      • 이미지나 용량이 큰 파일들을 사이트에 한번 접속해서 다운받은 다음 재접속시 다시 받아올 필요가 없도록 데이터가 저장되는 것

      • 저장위치는 캐시의 종류에 따라 다름

    쿠키, 세션 : 특정 정보 기록

    캐시 : 시간과 통신데이터를 절약하기위해 한번 로드한 데이터를 뒀다가 재사용 하는 것

    이벤트/정보 수집은 client 단에서 일어남

    보통 웹에서는 GA(google analytics)에 맡기고, 앱은 firebse를 많이 사용한다

    DB

    특징을 알아보자.

    예시: 0.00001 퍼센트의 확률로 고객의 돈이 날아가는 은행이 있다 $\rightarrow$ 아무도 안씀

    데이터의 무결성

    예시: 1번부터 10000번까지의 데이터를 조사해서 수정하는 시간이 1초라면, 1억개의 데이터가 있다면 데이터 하나 수정하는데 몇시간이 걸림 $\rightarrow$ 장바구니 넣는데 5초 걸린다면 아무도 안씀

    빨라야 함

    잡담 : 실무에서, 데이터베이스 구조는 잘 안바꾸려함
    데이터베이스 수정시 오래걸리고(테스트 많이해야함), 신중해야함
    데이터베이스는 빈번하게 바꿀 순 없음 (문제가 생기면 큰일 이니까)

    관계형 데이터 베이스(RDB)

    Relational database : DB 특징을 만족하기 위해 개발된, 현재의 가장 많이 사용되는 방법론

    관계형 데이터 베이스의 철학을 바탕으로 만든 문서 작업 툴이 바로 엑셀임

    DBMS

    • RDBMS(Relational database management system) : MS SQL Server, Oracle Database, MySQL, MariaDB

    OS(Ubuntu) - 서버 프로그램 - 한 두개의 DBMS

    SQL(Structured Query Language) : DBMS 에 요청하는 명령어

    Sever, Client

    • DBMS 는 메모리(RAM)위에서 돌아가서 (CPU가 컨트롤) 하드디스크로 데이터를 저장시킴

    • 즉, CPU, 메모리, 보조기억장치만 있으면 DBMS 사용가능

      • 예시 : 핸드폰 알람은 비행기 모드에서도 울림

    서버에 있는 데이터는 서버 개발자가, 클라이언트에 있는 개발자가 다룸

    클라이언트 데이터 : 로컬, 내부DB, 네이티브, 클리/클라이언트, 프론트/프론트앤드

    서버 데이터 : 서버, API 요청, DB, 백/백앤드
    (GET 으로 가져온거)

    이미지데이터

    이미지는 주소를 가지고 있고, 클라/서버/다른 곳 어디 있는지에 따라 나뉨

    직무분류


    • 웹

      1. 웹페이지 접속
        • 웹 클라이언트, 웹 프론트앤드 : 특정 주소로 접근했을 때, 전달되는 HTML, CSS, JS 파일 만듦 (+ 그걸 전달해주는 서버프로그램 설치)
      2. 검색 등 기능 사용 (서버로 API 요청, 응답 받아서 웹페이지를 띄움)
        • 서버/백앤드 : 서버 위에 서브 프로그램을 코딩해서 올려놔야함. DBMS 연결.
          • 예시 : 스프링으로 API 모듈 만들기 (트래픽 몰릴 때 처리하는 코드)

    • 앱

      1. 앱 다운로드
        • 모바일 클라이언트 혹은 모바일 프론트앤드 : 앱 만들어서 스토어에 올려놔야함

    • 관리자 필드 (back office) : 보통 웹으로 많이 만들음 (서버 또 필요)

      • 웹 클라이언트, 웹 프론트앤드 : 관리자가 접속하면 서버에 내용을 웹으로 띄어줌 (그 웹 페이지를 만들고, 서버프로그램 설치 등)

      • 관리자 : 전달받은 웹으로 서버 컴퓨터로 접속해서 관리 (재고 관리 등)

    서버가 여러개인 나누는 이유 : 컴퓨터 고장시 대비

    • 웹의 기본 : HTML, CSS, Javascript
    • 웹 클라이언트 라고 부르는 이유 : 서버의 원본 파일을 다운 받아서 표현하기 때문
    • 브라우저

    웹의 기본

    • HTML (HyperText Markup Language) : 서로 다른 시스템과 편집툴을 거쳐도 공용적으로 사용할 수 있도록 문서타입을 만든 것 (즉, 지식을 공유하기 위한 문서작업이 발단이 됨)
    • CSS : HTML에 디자인 개념을 넣은 것
      • HTML + CSS => 퍼블리싱(퍼블리셔)
    • Javascript : 기능 구현
      • 웹으로 API 요청 + 서버의 기능 사용 … 데이터베이스 사용… 등 다이나믹한 모습을 원하게 됨

    크롬 개발자 도구 웹 이해하기

    검색을 했을 때 이루어지는 요청과 응답 과정을 확인해 보자!




    1. a 검색 (=> request)

    2. respond 한 걸 (json형식), Javascript 가 가져옴

      • 크롬 개발자 도구의 Network 탭에서 HTTP 상태코드 확인 가능

        • 200 => 응답성공 (= 웹 페이지가 뜸/표시됨)

        • Network > Response 탭에서 json 형식 응답이 옴

    3. 우리가 아는 예쁜 화면을 표시, Javascript 처리함

      • html(문서), css(디자인)만으로 불가. 즉, javascript가 처리해줘야함

    이 모든건 서버의 원본 파일을(HTML, CSS, Javascript 원본), 클라이언트에서 받아온 것

    • 앱 - 설치하여, 실행파일로 계속 실행
      • [단점] 수정을 위해서는, 설치파일을 업데이트 한 후, 사용자의 업데이트가 필요
      • [장점] 네트워크의 영향을 덜 받음
    • 웹 - 그때그때 필요한 파일을 서버로 부터 받아옴
      • [장점] 수정시 원본을 바꾸면 바뀜 (새로고침을 하면 바뀜!)
      • [단점] 네트워크에 따라 달라짐
        • 극복하기 위한 시도 (PWA) 등이 존재

    • API

      • API : 공공 또는 특정그룹에게 공개되서 정보를 읽거나 넣거나 변경할 수 있는 자판기 같은 것

    : 자바스크립트 라이브러리, 가상 돔을 채용하여 대중화

    jQuery

    : 오픈소스 기반의 자바스크립트 라이브러리

    • (웹사이트에 자바스크립트를 쉽게 활용할 수 있도록 도와줌)
      • jQuery.ajax() 메서드 유명함
    • 사용 하락세
      • 웹 표준 API의 확장(ex : Fetch API)
      • Bootstrap의 제이쿼리 의존성 제거 등
      • 웹브라우저 환경 변화 (인터넷 익스플로러가 점유하던 시절갔고, 크롬 등 브라우저 사용됨(윈도우, 맥(Mac), 리눅스(Linux) 등 여러 OS 지원하기 때문) = 우수한 성능의 렌더링 엔진 탑재 + 웹표준의 신속한 반영)
      • Virtual DOM 사용하는 라이브러리 (리엑트, 뷰) 등장
