    네트워크 정의

    네트워크 : 분산되어있는 컴퓨터들이 자원을 공유할 수 있게 통신망으로 연결한 것

    연결이 필요한 곳 - 데이터베이스, 스토리지, 서버 등

    네트워크의 중요성

    네트워크는 4차 산업혁명 요소 기술들과 긴밀하게 연결

    • 3차 산업혁명

      • 디지털 인터넷(공유) = 지식 정보화 사회 → 급격한 기술의 발전

    • 4차 산업혁명

      • 정보통신 기술의 융합 - 대표요소 기술로는 빅데이터, 인공지능, 사물 인터넷 등 7 가지

      • On network - 대부분의 요소 기술들은 네트워크 위에서 동작 (IoT)

      • 블록체인 기술 - 신뢰 가능한 인터넷 발생 가능성

    글로벌 기업과 대기업의 중요한 인프라 기술

    • FAANG

      • Facebook, Apple, Amazon, Netflix, Google :Network 연결에서 동작하는 서비스 제공 기업

      • 구글 GCP : 클라우드 서비스 제공, 서울에 데이터 센터 구축

        (Cloud 서비스 : 가상화된 네트워크 서비스, Amazon의 Amazon Web Service)

    • 삼성 : 한국에서 세계최초 모바일 5G 서비스 상용화

    • 현대 : 제조에서 ICT 기업으로, 자율주행차 투자

    • LG : 자율주행 및 클라우드 서비스

    네트워크의 형태

    물리적 네트워크 LAN, WAN

    • LAN(Local Area Network) 근거리 통신망

      • 사무실 또는 학교 등의 가까운 지역을 한데 묶은 네트워크

    • WAN(Wide Area Network) 장거리 통신망

      • 각각 떨어진 LAN 망을 연결, 규모가 큰 네트워크, ISP 로 연결

    • WLAN(Wireless Local Area Network) 무선(거리) 통신망

    논리적 네트워크 VPN

    • VPN(Virtual Private Network) 가상 사설망
      • 공중망을 사설망처럼 사용, 암호화
        • 예시) 학교 내에서는 서로 통신 가능, A LAN회사 B LAN 회사 끼리는 통신 불가함 (통신하려면 라우팅 필요) 그러나 논리적으로 외부에 있더라도, 그 회사의 LAN 처럼 쓰겠다 하는게 VPN

    네트워크 표준 기구

    • ISO 국제 표준화 기구
      • OSI 7 Layer -> ISO Standard Model
    • IEEE 미국전기전자협회 (주로 LAN)
      • 이더넷 : IEEE 802.3
    • ITU-T 국제전기통신연합 통신표준본부 (주로 WAN)

    인터넷 표준 기구

    • IETF 인터넷 엔지니어 테스크포스

    • RFC - 프로토콜 정의 문서 (모든 통신에는 프로토콜이 정의됨)

      • TCP/IP : RFC 1122 & 1123
      • HTTP/1.1 : RFC 2616

    네트워크 구조

    네트워크 구조는 크게 규모 , 서비스 , 통신방식, 구현방식 등을 통해 설계 된다.

    • 규모 (PC 대수, 사용 데이터량에 따라)

      • 회사나 학교 등의 집단 크기에 따라 구분 - 사용자, 대역폭

    • 업종 (어떤 서비스가 중요한지)

      • 공공기관 , 제조, 금융, 게임 등의 업종에 따른 서비스 중요도

    • 통신 방식과 경로

      • Server & Client (웹사용)
      • Peer to Peer (PC 간의 통신)
      • 업체별로 커스터마이징 됨

    • 토폴로지

      • Star, Ring, Mesh, Bus, Tree, Redundancy

        동그라미: 컴퓨터, 네모: 서버

        보통 물리적네트워크 사용시 가장 많이 이용
        서버, 장비 등 다양하게 사용됨

    네트워크 형태

    네트워크는 기본적으로 홈, 기업, 클라우드 형태가 있다.

    홈네트워크

    인터넷 - isp - 모뎀(한대만 연결 가능) - 공유기 - 컴퓨터

    기업용 네트워크

    ISP - 전용선 - 라우터 - 방화벽 - L3 백본 - L2 스위치 - 서버, 컴퓨터 - L4 로드밸런서 (서비스배치함) - DMZ

    클라우드 네트워크 - AWS 기준

    인터넷 - Route53 - IGW - VPC - ELB - Auto Scaling - Securtiy Group - EC2

    • 인터넷 게이트 웨이나 모듈로 들어옴
    • Route53 : 인터넷 웹페이지 접속을 위해 입력하는 url은 사람이 읽기 좋은 string 임. 이걸 IP(숫자)로 바꾸는 것을 DNS 라 하는데 여기선 Route53 사용
    • VPC : AWS의 VPC로 기존의 L3 스위치나 L2 스위치, 라우팅 같은 네트워크 설계를 하나의 논리적인 망으로 구성한 것
    • ELB : VPC 를 통해 들어온 데이터들은 ELB( Elastic Load Balancing)를 통해 각각의 서버로 분산됨
    • Auto Scaling : 네트워크 대역폭과 서버의 저장 용량을 자동으로 조절
    • Securtiy Group : 기존의 방화벽 역할
    • SG : 스토리지
    • EC2 : 서버나 컴퓨터를 의미

    OSI 7 Layer

    네트워크 프로토콜과 통신을 7 계층으로 표현한 표준 모델

    • 목적
      • 프로토콜을 기능별로 나누고 계층별로 구분
      • 벤더간 호환성을 위한 표준 필요 -> 쉬운 접근성을 위해 기술의 발전
    • 역사
      연도 내용
      1970년대 초 네트워크는 정부 또는 특정 벤더에서 독점 개발, 공개형 모델의 필요
      1970년대 말 ISO (국제 표준화 기구) 에 의해 관리
      1984년 ISO 7498 릴리즈(발표)

    OSI 7 Layer 모델

    • OSI 7 Layer 7 계층 도식화

      OSI 7 Layer 모델 단계 설명
      $\uparrow$ Application 응용 서비스 HTTP(웹), SMTP(메일)
      사 Presentation 인코딩, 암호화, 압축
      람 Session TCP/ IP 통신 연결을 수림, 유지, 중단
      Transport TCP / UDP
      기 Network IP 통신, 라우팅
      계 Data Link 이더넷, 랜카드 , MAC 통신, 에러 검출/ 재전송
      $\downarrow$ Physical 네트워크 하드웨어 전송기술

    이제 각 계층의 기능을 알아보자!

    1계층 - Pysical

    • 장치와 통신매체 사이의 비정형 데이터의 전송을 담당

    • 디지털bit(0&1)을 전기, 무선 또는 광신호로 변환

    • 전송되는 방법, 제어 신호, 기계적 속성 등을 정의

    • 장비 : 케이블, 인터페이스, 허브, 리피터 등

      케이블 인터페이스 허브

    2계층 - Data Link

    • 동일 네트워크 내에서 데이터 전송, 링크를 통해서 연결을 설정하고 관리

    • 물리 계층에서 발생할 수 있는 오류를 감지하고 수정

    • IEEE 802 에서 정의

      • MAC (media access control) ,LLC( Logical Link Control)

        : 각각의 장비들은 고유한 코드를 갖고 통신 하겠다

    • 장비 : 모뎀, 스위치

      모뎀 스위치

    3계층 - 네트워크 계층

    • 다른 네트워크로 데이터 전송 , IP (internet protocol) 주소로 통신

      • 팀팀 -> 회사 확장

    • 출발지 IP 에서 목적지 IP 로 데이터 통신시, 중간에서 라운팅 처리

    • 데이터가 큰 경우 분할 및 전송 후 목적지에서 재조립하여 메시지 구현

    • ip통신과 라우팅

    • 장비 : L3 스위치와 라우터

      백본 L3 스위치 라우터

    4계층 - Transport

    • 호스트 간의 데이터 (서비스) 전송

      • 포트를 정해줘서 메일, 웹 등을 지정

    • 오류 복구 및 흐름 제어, 완벽한 데이터 전송을 보장

    • TCP/ UDP

    • L4 계층을 특정 하드웨어로 구분하기 모호

    • 장비 : L4 로드밸런서(Port 를 제어한다는 의미임)

      L4 로드밸런서

    5계층 - Session

    • 로컬 및 원격 애플리케이션 간의 IP, port 연결을 관리

      • Session Table 에서 관리함

    캡슐화

    종류 2가지 : 인캡슐레이션, 디캡슐레이션

    인캡슐레이션

    : OSI 7계층 (Application layer) 에서 하드웨어(Physical layer)까지 어떻게 통신이 되는지에 대한 내용

    하나씩 하나씩 데이터를 캡슐화 하고, 헤더를 붙이는 단계

    1. Host Data 요청 (Application layer)

      : 사용자가 컴퓨터에 접속을 하고, 어떤 데이터(ex. 사진)에 대해 상대방 컴퓨터에 통신을 하고자 함

    2. Data 로 표현 (Presentation layer)

      : 컴퓨터가 이해하는 일련의 정형화된 데이터로 표현

    3. TCP header + Data (Transport layer)

      : TCP header에 우리가 보내는 데이터가 어떤 서비스, 포트를 쓰는지 지정

      • transport layer 에선 데이터를 segment 라 부름

    4. IP header + Data (Network layer)

      : 어떤 목적지 IP 인지 + 기존 데이터를 인캡슐해서 다음 계층으로 보냄

      • Network layer 에선 데이터를 Packet 이라 부름

    5. MAC LLC header + Data + FCS (Data layer)

      : MAC주소(네트워크 장비의 물리적 시리얼 번호)가 들어있는 MAC LLC header 를 붙이고,
      오류 검출을 위해 FCS 라는 트레이를 붙임

      • Data layer 에선 데이터를 Frame 이라 부름

    6. 전기신호(Signal)로 변화 (Pysical layer)

      • Pysical layer 에선 데이터가 bit로 표현 됨

    디캡슐레이션

    : Physical에서 Application까지

    인캡슐레이션과 반대로 하나하나 떼내며 진행

    1. 전기신호(Signal)가 물리적 장비를 통해 들어옴 (Pysical layer)

      • bit

    2. MAC LLC header + Data + FCS (Data layer)

      • Frame
      • Mac header가 붙고, Mac Address를 보고 signal이 가야할 하드웨어(랜어댑터) 쪽으로 전달

    3. IP header + Data (Network layer)

      • Packet
      • Mac header는 이제 필요없으니 떼어냄, ip 정보를 보고 찾은 정보를 transport 로 올림

    4. TCP header + Data (Transport layer)

      • (여기부터 컴퓨터) TCP header 헤더를 보고 어떤 포트를 사용하는지를 정의
      • ip 헤더 날림 (이제 확인했으니까)

    5. Data 로 표현 (Session / Presentation layer)

      • 데이터 형성 됨

    6. Host Data 요청 (Application layer)

      • 사용자가 볼 수 있는 어플리케이션으로 호스트에게 넘어감
        • 데이터 $\rightarrow$ host data

    7. 사용자 PC 로 수신 (컴퓨터)

    모든 데이터는 네트워크 통신을 하게 되어있고,
    네트워크는 캡슐화를 통해 통신한다.

    CSMA / CD (Carrier sense multiple access/collision detection)

    • 송신 노드는 데이터를 전송하고, 다음 채널에서 다른 노드의 데이터 충돌 발생을 계속 감지

    • 충돌 발생 시, 모든 노드에게 충돌 발생을 통지하고 재전송을 시도

    • 동작 순서

      1. Carrier Sensing : 데이터를 보내기 전에 다른 노드에서 데이터를 보내는 중인지 확인
      2. Multiple Access: 데이터 보내느 곳이 없다면 전송 시작
      3. Collision Detection: 동시간대에 데이터를 보내게 되면 충돌이 일어나고 정지
      • 그 이후 특정시간이 지나면 다시 첫번째 단계로 반복

    이러한 CSMA / CD 통신을 허브의 Half Duplex(반이중 전송 방식) 이라 함

    전송방식

    • Simplex : 단방향 통신으로 수신측은 송신측에 응답 불가
    • Half Duplex : 반이중 전송방식으로 양방향 통신이나 송수신 시간은 정해짐 (무전기)
    • Full Duplex : 전이중 전송방식으로 동시 양방향 통신이 가능 (전화기)

      - 멀티모드(Multi Mode) - 다양한 경로나 모드를 가지고 전송한다. - 멀티모드는 주황색 케이블이다. - 싱글모드에 비해 비교전 단거리(2km이내)의 장비 간 전송에 사용된다. - 이중파장대(850nm(단파장), 1300nm(장파장))에 최적화

    Performance 부분 중요

    bit & Byte

    2진수는 Binary(0, 1)로 이루어지며 True & False 등 신호를 표현, 이것을 1bit 라 함

    • 1Byte = 8bit

    • 일반적인 사용

      • bit $\rightarrow$ Speed
        • 100Mbps = 100 Mbps(Mega bit per second)
      • Byte $\rightarrow$ Data Size
        • SDD 50GB = 50 Giga byte

    • 단위 : Kilo, Mega, Giga, Tera

      • k $\rightarrow$ M $\rightarrow$ G $\rightarrow$ T 로 넘어갈 때,
        10의 3제곱 1000 이 아닌 2의 10제곱 1024 (전산학에서는 2진수 기반이기 때문에)
        • 1024k = 1M, 1024M = 1G, 1024G = 1T

          Bandwidth, Throughput, BackPlane

    • Bandwidth(대역폭) : 주어진 시간대에 네트워크를 통해 이동할 수 있는 정보의 양

    • Throughput(처리량) : 단위 시간당 디지털 데이터 전송으로 처리하는 양

      대역폭이 8차선 도로라면, 처리량은 그 도로를 달리는 자동차의 숫자(양)과 같음

    • BackPlane : 네트워크 장비가 최대로 처리할 수 있는 데이터 용량

    CPS, CC, TPS

    장비 계측 일반적인 성능 지표 3가지

    • CPS(Connection Per Second) : 초당 커넥션 연결 수, L4(Session 단계에서 다시 설명)
    • CC(Concurrent Connections) : 초당 수용 가능한 커넥션
    • TPS(Transactions Per Seconds) : 초당 트랜잭션 연결 수, L7, 주로 HTTP 성능

    데이터 링크 계층과 스위치

    데이터 링크 계층의 역할과 기능

    Data Link Layer 역할

    • OSI 7 Layer의 2계층으로 인접한 네트워크 노드끼리 데이터를 전송하는 기능과 절차를 제공

    • 물리계층에서 발생할 수 있는 오류를 감지하고 수정

    • 대표적인 프로토콜 : 이더넷

    • 대표적인 장비 : 스위치

      데이터 단위가 Frame 이며, Layer 3로 올라가면 packet 단위로 바뀜

    Data Link Layer 구성

    • 2개의 부 계층으로 구성

      • MAC(Media Access Control)
        물리적인 부분으로 매체간의 연결방식을 제어하고 1계층과 연결 (물리적 인터페이스)
      • LLC(Logical Link Control)
        논리적인 부분으로 Frame을 만들고 3계층과 연결

    데이터 링크 계층은 LLC와 MAC으로 구성되어 있고 Frame 단위로 통신한다
    상위 계층은 Packet 단위로 통신한다

    MAC 주소

    • 확인방법 : CMD ipconfig/all 또는 네트워크 설정에서 확인

      CMD창 ipconfig/all 네트워크 설정 확인
      • 48bit(6 Bytes) 6자리로 구성, 16진수로 표현
      • 앞에 3자리는 OUI(제조사 식별 코드)
        ex) 76:b5:87
      • 나머지 3자리는 제조사 내 일련번호
        ex) 57:3a:20

    Data Link Layer 주요 기능

    Framing, 회선제어, 흐름제어, 오류제어 등이 있다.

    Framing

    : 데이터그램을 캡슐화하여 프레임 단위로 만들고 헤더와 트레이러를 추가

    • 헤더 : 목적지, 출발지 주소 그리고 데이터 내용을 정의

    • 트레일러 : 비트 에러를 감지

      • 상위 layer 인 Network(3계층)에서 Data Link로 내려올 땐, Packet 이 Framing(프레임화) 되면서 header 와 trailer 가 붙음
      • 또는 하위 layer 인 Pysical(1계층)에서 bit signal을 Frame 단위로 바꿔주는 것 역시 Framing(프레임화)라 함

    회선제어

    : 신호간의 충돌이 발생하지 않도록 제어 (통신 경로 제어)

    기법 - ENQ/ACK 방법 (1:1), Polling 방법 (1: n)

    • ENQ/ACK 방법
      : 전용 전송 링크 1:1

      • ENQ : 데이터 전송에서 상대국의 응답을 구하는 전송제어 부호.
      • ACK : 송신 측에 대하여 수신 측에서 긍정적 응답으로 보내지는 전송 제어용 캐릭터
      • EOT(End of Transmission Character) : 전송 종료 문자

    • Polling 방법 (1: n)
      : ENQ/ACK 방법과 다르게 동일한 패킷을 동시에 여러 PC에게 전송 가능

      • 2가지 모드
        • Select 모드 : 송신자가 나머지 수신자들을 선택하여 전송

          여러 PC에 동일한 패킷 전송 가능

        • Poll 모드 : 수신자에게 데이터 수신 여부를 확인하여 응답을 확인하고 전송 (multipoint)

          필요한 PC에게만 보냄

    흐름제어(Flow control)

    : 송신자와 수신자의 데이터를 처리하는 속도 차이를 해결하기 위한 제어

    Feedback 방식의 Flow Control (Layer 2에서), 상위 계층(Layer 3, 4, …)은 Rate 기반

    예시 : 1G TPS 대용량 컴퓨터 A와 10M TPS 짜리 PC B를 같이 사용한다고 가정해보자. A에서 1G를 B에게 보내주면 B는 당연히 처리를 못한다. 이런 속도 차이를 맞춰줘야 처리를 할 수 있다. PC B가 처리할 수 있는 만큼만 보내는걸 Flow Control 이라고 한다.

    기법 - STOP & WAIT, Sliding window

    • STOP & WAIT

      • 데이터를 보내고 ACK 응답이 올때까지 기다림

        • ACK(Acknowledgement)

      • 간단한 구현, 비효율적(기다려야 하니까)

      • Time-out Interval
        : Error 발생할 경우 대처법, Retransmit frame 사용
        예시 : Frame 전달 후 ACK가 회선 문제로 응답하지 않을 경우

        • 데이터를 보내고 ACK 응답이 올 때 까지 기다림
        • Retransmit frame : ACK 오지 않으면 일정시간 후 다시 보냄

      • Duplicate frame
        : Frame을 재전송시 발생 할 수 있는 문제로 Sequence number를 사용하여 해결 가능

        • Sequence number(1 bit)를 사용하여 동일 frame 인지 구분한 후 상위 계층으로 전달
        • 시퀀스 넘버를 확인하여 중복 데이터

    • Sliding window (개선된 방식)

      • ACK 응답 없이 여러 개의 프레임이 연속적으로 전송 가능

      • Window size 만큼 송수신을 함 (buffer가 넘칠 시 문제 발생)

        • Window size : 전송과 수신측의 데이터가 저장되는 버퍼의 크기 (커졌다 작아졌다, 빨라졌다 느려졌다 함)
        • Window size = (현재 ACK) - (이전에 보낸 ACK)

    오류 제어

    : 전송 중에 오류나 손실 발생 시 수신측은 에러를 탐지 및 재전송

    ARQ(Automatic Repeat Request) : 프레임 손상 시 재전송이 수행되는 과정

    • Stop & Wait ARQ
      : Flow control에 ARQ가 추가, Frame 손상 시 NAK 을 보냄

      • 전송 측에서 NAK을 수신하면 재전송
      • 주어진 시간에 ACK가 안오면 재전송 (Time-out Interval 까지 packet 이 안오는 경우)
    • GO BACK n ARQ (Sliding window 방식의 하나)
      : 여러개 Frame을 (묶음으로) 전송 가능
      • 전송측 Frame 6개 전송
      • 수신측 3번 Frame 문제 발생, Frame 345 폐기
        • 비효율적인 부분 - 3만 이상한데, 345 모두 폐기
      • 수신측 NAK 3으로 손상 응답
      • 전송측 3번이 포함된 345 재전송
      • 재수신
    • Selective Repeat ARQ
      : 손상된 Frame만 선별해서 재전송

    이더넷 프레임 구조

    • Ethernet v2
      : 데이터 링크 계층에서 MAC(Media Access Control) 통신과 프로토콜의 형식을 정의

      • FCS(Frame Chech Sequence) : 에러 체크

    스위치

    : 2계층의 대표적인 장비로 MAC주소 기반 통신

    • 배경 : 허브의 단점을 보완

      • 보완 1 : Half Duplex $\rightarrow$ Full duplex (동시 양방향 통신으로)
      • 보완 2 : 1 Collision Domain $\rightarrow$ 포트별 Collision Domain (불필요한 곳에는 패킷을 돌리지 않음)

    • 라우팅 기능이 있는 스위치 = L3 스위치 라고도 불림


      가운데 UDP, 오른편 SDP 꼽는 곳

    스위치 동작방식

    목적지 주소를 MAC 주소 테이블에서 확인하여 연결된 포트로 프레임 전달

    Learning - Flooding - Forwarding - Filtering - Aging

    1. Learning
      : 출발지 주소가 MAC 주소 테이블에 없으면 해당 주소를 저장

      • 4개의 PC는 스위치의 각 포트에 연결되고 프레임이 스위치에 전달
      • 스위치는 해당 포트로 유입된 프레임을 보고 MAC 주소를 테이블에 저장
        => 연결된 PC의 정보는 다 저장함

    2. Flooding (Broadcasting)
      : 목적지 주소가 MAC 주소 테이블에 없으면 전체 포트로 전달
      : 스위치가 목적지 주소를 모를 경우 Broadcasting

      • PC1은 목적지가 aa:bb:cc:dd:ee:05 주소로 프레임 전달
      • 스위치는 해당 주소가 MAC Table에 없으면 전체 포트로 전달

    3. Forwarding
      : 목적지 주소가 MAC 주소 테이블에 있으면 해당 포트로 전달

      • pc1은 목적지 aa:bb:cc:dd:ee:05 주소로 프레임 전달
      • 스위치는 해당 주소가 MAC Table에 존재하므로 해당 프레임을 PC5로 전달

    4. Filtering - Collision Domain
      : 출발지와 목적지가 같은 네트워크 영역이면 다른 네트워크로 전달하지 않음

      • Collision Domian을 분리, 구분 가능 (=Filtering 이 가능하다) 하여 효율적 통신이 가능

      • pc1은 목적지 aa:bb:cc:dd:ee:02 주소로 프레임 전달

      • 스위치는 해당 주소가 동일 네트워크 영역임을 확인하여 다른 포트로 전달하지 않음

    5. Aging
      : MAC 주소 테이블의 각 주소는 일정 시간 이후에 삭제

      • 스위치의 MAC 주소 테이블은 시간이 지나면 삭제
        • 테이블 저장 공간의 효율적 사용을 위함
      • 해당 포트에 연결된 PC가 다른 포트로 옮겨진 경우도 발생
      • 기본 300초(Cisco 기준) 저장 다시 프레임이 발생되면 다시 카운트

    스위치 동작 과정 (정리)

    1. 프레임 유입
    2. 출발지 주소가 MAC 주소 테이블에 있는지 없는지
      • 경우 1 (없음 - 신규) : 출발지 주소 저장 $\rightarrow$ Learning
      • 경우 2 (있음 - 기존) : Refresh $\leftarrow$ Aging 타이머 재시작
    3. 목적지 주소가 MAC 주소 테이블에 있는지 없는지 + 같은 포트에 있는지
      • 경우 1 (없음) : 모든 포트에 전달 (Broadcasting) $\rightarrow$ Flooding
      • 경우 2 (있음) : 해당 포트에 전달 $\rightarrow$ Forwarding
      • 경우 3 (같은 포트) : 다른 포트에 전달할 필요가 없음 $\rightarrow$ Filtering|

    ARP(Address Resolution Protocol)

    : IP주소를 통해서 MAC 주소를 알려주는 프로토콜

    • 배경 : 컴퓨터 A가 컴퓨터 B에게 IP 통신을 시도하고 통신을 수행하기 위해서는 목적지 MAC 주소를 알아야 함

      목적지 IP에 해당되는 MAC 주소를 알려주는 ARP 탄생

      ① Network에서 통신시도를 하면 하위계층(Data Link)로 넘어감
      ② 이때, PC2의 MAC 주소를 모르기 때문에 ARP Request 를 통해 PC2 에게 주소를 물어봄
      ③ PC2가 자신의 MAC 주소를 ARP Reply 를 통해 알려줌

      ARP는 목적지의 MAC 주소를 (요청-응답하여) 알려주는 역할

    PC1은 동일 네트워크 대역인 목적지 IP 172.20.10.9 로 패킷 전송을 시도

    1. 목적지 MAC 주소를 알기 위해서 우선 자신의 ARP Cache Table을 확인

      • CMD 에서 arp -a 명령어로 확인 가능

        172.20.10.9는 없으므로 ARP Request 수행

    2. ARP Cache Table에 있으면 패킷 전송, 없으면 ARP Request 전송 - Broadcasting

      • WireShark 에서 패킷덤프 확인 가능

        arp 검색
        Who has 172.20.10.9? Tell 172.20.10.2 라고 MAC 주소 요청
        Broadcast 날림 (보통 마지막 주소(ff:ff:ff:ff:ff:ff)를 브로드캐스트 주소라고 함)

    3. IP 172.20.10.9 에서 목적지 MAC 주소를 ARP Reply 로 전달

      • WireShark 에서 패킷덤프 확인 가능

        172.20.10.9 is at 76:b5:87:75:83:64 라고 MAC 주소를 PC1에 전달 (ARP Reply로)
        Destination과 Source 맥이 다 보이는걸 확인 가능

    4. 목적지 MAC주소는 ARP Cache Table에 저장되고 패킷 전송

      • CMD 에서 arp -a 명령어로 확인 가능

        172.20.10.9는 없으므로 ARP Request 수행

    스패닝트리 프로토콜과 루핑

    Looping

    : 같은 네트워크 대역 대에서 스위치에 연결된 경로가 2개 이상인 경우에 발생

    • 예시
      • PC가 브로드캐스팅 패킷을 스위치들에게 전달하고 전달 받은 스위치들은 Flooding을 함
      • 스위치들끼리 Flooding 된 프레임이 서로 계속 전달되어 네트워크에 문제를 일으킴
    • 발생 원인 : 회선 및 스위치 이중화 또는 증축 등

      물리적인 포트 연결의 실수 또는 잘못된 이중화 구성으로 L2에서 가장 빈번히 발생하는 이슈

    Looping 구조

    1. PC1은 Switch 1에게 브로드캐스팅 전송
    2. Switch 1은 모든 포트에 브로드캐스팅 전송
    3. 전달받은 브로드캐스팅 프레임을 Switch 2, 3도 모든 포트에 전송
    4. Switch 1은 Switch 2, 3 에게 다시 전달 받은 브로드캐스팅을 다시 모든 포트에 전송

    이러한 문제점이 발생하는 것이 Looping이다
    (브로드캐스팅이 엄청 많이 발생했다해서 Broad casting Storm 이라는 논문도 나옴)

    STP(Spanning Tree Protocol)

    : 루핑 방지를 자동으로 하기 위한 프로토콜 (스패닝 트리 알고리즘 사용)

    스패닝 트리 알고리즘 : 자동으로 루핑을 막아주는 알고리즘 (필요없는 경로를 격리, 고립 시킴)

    • 배경 : Looping의 문제점을 해결

    • IEEE 802.1d 에 명시됨

    • 두가지 개념

      1. Bridge ID
        : 스위치의 우선순위를 0~65535로 설정 (낮을수록 우선순위가 높음)

      2. Path Cost
        : 스위치 간의 경로가 어디가 더 빠른지 계산

        • 링크의 속도(대역폭), 1000/링크 속도로 계산됨 (작을수록 우선순위가 높음)
        • 그러나, but 1Gbps속도가 나오면서 계산법이 적합하지 않게 됨 $\rightarrow$ IEEE에서 각 대역폭 별 숫자를 정의
          • 10Mbps = 100, 100M bps = 19, 1Gbps = 4 (가중치라 생각하면 됨, 외우면 좋음)

            STP의 요소

    Root Bridge, Root Port, Designated Port, Path Cost가 있음

    1. Root Bridge를 네트워크당 1개 선출
    2. Root Port 선출 (Root Bridge가 아닌 스위치들에서)
    3. Designated Port 지정 (각 세그먼트별로 1개 씩)

    Root Port와 Designated Port는 Root Path Cost로 선출 됨
    (Root Port 는 가장 작은 Root Path Cost, Designated Port 는 루트 브리지 ID > Path Cost > 브리지 ID > 포트 ID 우선순위로 결정 됨)

    BPDU(Bridge Protocol Data Unit)

    : 스패닝 트리 프로토콜에 의해 스위치간 서로 주고받는 제어 프레임

    • 배경 : STP 를 만들기 위해 요소들은 선출하기 위해서, 각각 통신할 수 있는 통신수단이 필요

    • 종류

      1. Configuration BPDU (구성 관련된 부분을 업데이트 해줌)
        : 루트 브리지에 의해 생성

        • Root BID – 루트 브리지로 선출될 스위치 정보
        • Path Cost – 루트 브리지 까지의 경로 비용
        • Bridge ID, Port ID – 나머지 스위치와 포트의 우선순위

      2. TCN(Topology Change Notification) BPDU (구성이 변경될 때, 업데이트가 필요하므로)
        : 네트워크 내 구성변경시 통보 (거의 2초당 한번씩 발생)

        • 우선순위 매기는 기준 (낮은 숫자가 더 높은 우선순위를 가짐)
          • 누가 더 작은 Root BID?
          • 루트 브리지까지 더 낮은 Path Cost?
          • 연결된 스위치 중 누가 더 낮은 BID?
          • 연결된 포트 중 누가 더 낮은 Port ID?

    참고 : BPDU 관련 용어 해설

    STP 과정

    1. Root Bridge 선출

    1. 각각 스위치는 고유의 BID 를 가짐
      • BID : 2바이트(우선순위) + 6바이트 MAC 주소
        • 32768은 defalut 값 (0~65535사이 값이 올 수 있고, 그 절반값이 32768 이기 때문)
        • 각각 스위치의 MAC Adress는 0000.0000.1111, 0000.0000.2222, 0000.0000.3333
    2. 서로 BPDU를 교환하고 가장 낮은 숫자가 루트 브리지가 됨
      • Switch 1, Switch 2, Switch 3 순
    3. 우선순위 숫자는 명령어로 설정 가능

    2. Root Port 선출

    1. 나머지 스위치들은 루트 브리지와 가장 빠르게 연결되는 루트 포트를 선출
    2. 루트 포트는 가장 낮은 Root Path Cost 값을 가짐
    3. Switch 2는 P1 = 4 + 19, Switch 3은 P0 = 19
      • Switch 2에서 Root Bridge(Switch 0)로 가려면,
        • 방법1 (P0) : Switch 2 $\rightarrow$ Switch 1, Path Cost 는 10Mbps 로 100 의 가중치를 가짐
        • 방법2 (P1) : Switch 2 $\rightarrow$ 3 $\rightarrow$ 1, Path Cost 는 1Gbps + 100Mbps 로 23 가중치를 가짐 (Root Port)
      • Switch 3에서 Root Bridge(Switch 0)로 가려면,
        • 방법1 (P0) : Switch 3 $\rightarrow$ Switch 1, Path Cost 는 100Mbps 로 19 의 가중치를 가짐 (Root Port)
        • 방법2 (P1) : Switch 2 $\rightarrow$ 3 $\rightarrow$ 1, Path Cost 는 1Gbps + 10Mbps 로 104 가중치를 가짐

    3. Designated Port 선출

    • 각 세그먼트(링크)별 루트 브리지와 가장 빠르게 연결되는 포트를 Designated 포트로 선출

      • 우선순위는 루트 브리지 ID > Path Cost > 브리지 ID > 포트 ID
      1. 루트 브리지와 연결된 포트는 무조건 Designated Port 가 됨 (Switch 1 P0 & P1)
        • Switch 1 P0는 10Mbps 세그먼트의 Designated Port
        • Switch 1 P1은 100Mbps 세그먼트의 Designated Port
      2. 1Gbps 라인에서는 Switch 3 P1이 Designated Port로 선출
        • Switch 2 P1의 Path Cost : 100Mbps + 1Gbps = 104
        • Switch 3 P1의 Path Cost : 10Mbps + 1Gbps = 23 (Designated Port)

    STP 상태변화

    스위치의 포트는 스패닝 트리 프로토콜 안에서 5가지 상태로 표현
    Disabled - Blocking - Listening - Learning - Forwarding

    1. Disabled

      • 포트가 shut down 인 상태로 데이터 전송 불가, MAC 학습 불가, BPDU 송수신 불가

    2. Blocking

      • 부팅하거나 , Disabled 상태를 Up 했을때, 첫번째 거치는 단계. BPDU 만 송수신

    3. Listening - 15초

      • Blocking 포트가 루트 또는 데지그네이티드 포트로 선정되는 단계, BPDU 만 송수신

    4. Learning - 15초

      • 리스닝 상태에서 특정 시간이 흐른 후, 러닝 상태가 됨 MAC 학습 시작, BPDU 만 송수신

    5. Forwarding

      • 러닝 상태에서 특정 시간이 흐른뒤, 포워딩 상태가 됨, 데이터 전송 시작, BPDU 송수신

    예제

    • Looping 상태

      1. SW1 $\rightarrow$ SW2 경로는 2개

      2. SW2 $\rightarrow$ SW3 경로는 2개

      3. SW3 $\rightarrow$ SW1 경로는 2개

        Looping 상태 (계속 뺑뺑이 돌음)

    • BPDU 교환

      1. Switch 1 루트 브리지 선출

      2. 각 세그먼트 별 루트 & 데지그네이티드 포트 선정

        • R : Root , D : Designated , ND : Non-Designated (사용 x)

    • 상태 변화

      1. 루트 & 데지그네이티드 포트는 포워딩 상태, Non-데지그네이티드 포트는 블로킹 상태로 변화
        • Blocking : 이 경로로는 통신 불가

          논리적으로 고립된 구성으로 Looping 문제 해결

    RSTP & MST

    • RSTP (Rapid Spanning Tree Protocol) - 대부분
      • IEEE 802.1w 에 명시 됨
      • 배경 : STP를 적용하면 포워딩 상태까지 30~50초 걸림 (예를들어, 네이버가 처음에 뜨는데 30초 걸린다 생각하면.. 끔찍함)

        컨버전스 타임 단축(1~2초 내외) (Learining & Listening 단계가 없음)

    • MST (Multiple Spanning Tree)
      • IEEE 802.1s 에 명시 됨
      • 배경 : 네트워크 그룹이 많아지면 STP or RSTP BPDU 프레임이 많아지고 스위치 부하 발생

        여러개의 STP 그룹들을 묶어서 부하를 줄이고 효율적으로 관리

    VLAN (+Trunk)

    VLAN (Virtual LAN)

    • 물리적 구성이 아닌 논리적인 가상의 LAN을 구성하는 기술
    • 불필요한 데이터 차단: 브로드 캐스트 도메인 별로 나누어 관리
      • 하나의 스위치를 논리적으로 나눠서 스위치가 여러대인 것 처럼 사용
    • 관리의 용이성과 보안: 호스트의 물리적 이동 없이 LAN 그룹 변경이 가능
    • 비용 절감: 새로운 LAN 추가시 물리적 스위치 구매 필요없음

    VLAN 종류

    VLAN 구성방법은 Port, MAC주소, IP주소 기반이 있다

    1. Port 기반 VLAN
      • 여러개의 VLAN을 설정하고 각각의 LAN에 물리적인 포트를 지정
      • VLAN 변경이 필요한 호스트는 물리적인 포트 또는 스위치의 VLAN 설정을 변경
    2. MAC주소 기반 VLAN
      • 각 호스트 또는 네트워크 장비의 MAC 주소를 각각의 VLAN에 정의
      • 호스트가 이동되어도 VLAN 변경 필요 없음, 신규 호스트 연결시 설정 변경 필요
    3. IP주소 기반 VLAN
      • IP주소 서브넷 기반으로 VLAN을 나누는 방법 (IP도 하나의 브로드캐스트 영역이 됨)
      • IP(Internet Protocol): 3계층에서 사용하는 프로토콜
      • 서브넷: IP주소의 네트워크영역의 크기를 나눈것

    1. Port 기반 VLAN

    • fe0 ~ fe9 , Gi0 , Gi1 : 스위치 내부의 Port
    • 한개의 스위치를 Port를 기준으로 나눠서 LAN 구성
    • Table
      • VLAN 100 (총무팀) : fe0 ~ fe4
      • VLAN 200 (인사팀) : fe5 ~ fe7
      • VLAN 300 (영업팀) : fe8 ~ fe9
    • 변경시 VLAN 그룹에 매핑되어 있는 포트 설정 정보만 변경
    • 또는 물리적인 케이블을 이동하여 연결

    2. MAC 주소 기반 VLAN

    • Port에 상관없이 목적지의 MAC 주소에 따라 LAN을 나눠서 전송
    • Table
      • VLAN 100(총무팀): aa:bb:cc:dd:ee:11~20
      • VLAN 200(인사팀): aa:bb:cc:dd:ee:21~30
      • VLAN 300(영업팀): aa:bb:cc:dd:ee:31~40
    • VLAN 변경시 호스트의 MAC 주소를 다른 VLAN으로 이동
    • 신규 호스트 연결 시 MAC 주소를 확인하여 VLAN 그룹에 소속

    3. IP 주소 기반 VLAN

    • Port, MAC주소에 상관없이 목적지의 IP 주소에 따라 LAN을 나눠서 전송
    • Table
      • VLAN 100(총무팀): 192.168.1.0/24
      • VLAN 200(인사팀): 192.168.2.0/24
      • VLAN 300(영업팀): 192.168.3.0/24
    • VLAN 변경시 호스트의 IP 주소를 다른 VLAN으로 이동
    • 신규 호스트 연결시 IP 주소를 확인하여 VLAN 그룹에 소속

    Trunk

    : 물리적 스위치간 VLAN 연결 시 하나의 물리적 연결로 VLAN그룹들을 공유하는 기술

    • 물리적 스위치간 VLAN 연결

      • 대규모 망에서 스위치의 개수는 증가, VLAN 그룹 개수도 증가
        • 스위치 끼리 VLAN 그룹끼리의 통신이 필요하고 그에 따라 물리적인 연결이 필요 (100,200,300 연결 망)

          물리적 연결 케이블은 복잡

    • Trunk 연결 (하나의 물리적 연결)

      • 많은 수의 VLAN 그룹들도 물리적 연결 케이블은 1개로 구성

        하나의 물리적인 연결로 스위치간 VLAN 통신이 가능

    Trunk Protocol

    • 이더넷 프레임에 식별용 VLAN ID를 삽입하여 데이터를 구분하여 통신 및 제어 가능

    • IEEE 802.1q

    • VLAN Tagging: VLAN ID 정보

      • 하나의 물리적 연결로 통신하기 위해 VLAN을 구분해 놓는 것

    802.1q tagged format

    • 이더넷 프레임에 삽입되며 4바이트로 구성

    • TPID(Tag Protocol IDentifier): 태그되지 않은 프레임과 태깅된 프레임을 구별

    • TCI (Tag Control Information): 태그 제어 정보

      1. PCP(Priority Code Point): 프레임의 우선 순위
      2. DEI(Drop Eligible Indicator): 트래픽 혼잡시 제거되기 적합한 프레임들을 가리키는 용도
      3. VID(VLAN Identifier): VLAN이 어느 프레임에 속하는지 결정

    VLAN 구성

    VLAN 설계

    1. VLAN 그룹 정의
      • 총무팀 : vlan 100
      • 인사팀 : vlan 200
      • 영업팀 : vlan 300
    2. VLAN 구성방법 정의
      • 포트, MAC주소, IP주소
      • MAC 또는 IP주소 방식의 경우 미리 사전 조사 필요
    3. 트렁크 포트 정의
      • Throughput(대역폭) 확인
      • 허가(Tagged)할 프레임 정의, 정의 되지 않은 Tag는 통신 불가

    VLAN 설정

    VLAN 모드에는 access, trunk, dynamic 모드가 있음

    1. VLAN 그룹 설정

      # vlan 100
# vlan 200
# vlan 300

    2. 액세스 모드: 사용할 포트에 1개의 VLAN ID 설정

      # interface GigabitEthernet1/0/1
# switchport mode access
# switchport access vlan 100

    3. 트렁크 모드: 사용할 포트에 여러개의 VLAN ID 설정

      # interface GigabitEthernet1/0/2
# switchport mode trunk
# switchport trunk allowed vlan 100,200,300

    4. 다이나믹 모드: 연결된 포트들의 상태에 따라서 액세스 또는 트렁크로 변경되는 모드
