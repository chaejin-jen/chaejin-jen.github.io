    리눅스 시작

    • 리누스 토발즈(Linus Torvalds)가 개발
      • 대학에 있는 UNIX 컴퓨터를 집에서 쓰고 싶다.
      • 다중 사용자, 다중 작업(시분할 시스템, 멀티 태스킹)을 지원하는 유닉스(UNIX)와 유사한 운영체제

    GNU 프로젝트

    • GNU = Gnu is Not Unix
    • 유닉스 운영체제를 여러 회사에서 각자 개발, 소스를 공유하지 않는 문화에 반발
    • 리차드 스톨만: 초기 컴퓨터 개발 공동체의 상호협력적인 문화로 돌아갈 것을 주장하며, 1985 년도에 GNU 선언문을 발표
    • GNU 프로젝트를 지원하기 위해 자유 소프트웨어 재단(FSF) 설립과, GNU 공개 라이선스(GPL)라는 규
      약을 제공

    GPL 라이선스

    • GPL 프로그램은 어떤 목적으로, 어떤 형태로든 사용할 수 있지만, 사용하거나 변경된 프로그램을 다시 배

    • 포하는 경우, 동일한 GPL 라이선스로 공개를 해야 함

    • 소스 오픈을 장려하기 위함

    GNU 프로젝트와 GNU Hurd

    • 운영체제 커널 개발 시도 ‑ GNU Hurd

    • 운영체제에 필요한 라이브러리, 컴파일러, 에디터, 쉘 개발

      GNU hurd: https://www.gnu.org/software/hurd/

    • User $\rightarrow$ OS (자원요청)
      • User - Shell - Operating System - Hardware - Operating System
    • Application $\rightarrow$ OS (자원요청)
      • Application - Library or API (프로그래밍 언어별 API) - System Call - Operating System - Hardware - Operating System
    • 어떤 API를 만들까 해보니, 많이 쓰는 C언어를 사용했고, 컴파일러를 만들어야했고….
    • vi : 대표적인 유닉스 계열 에디터
      • 요즘엔 vim (vi improved)### GNU 프로젝트와 리눅스
    • GNU Hurd 개발 지연
      • 시스템콜, 컴파일러, 라이브러리와 API, 에디터, 쉘 개발했지만 OS가 빠짐
    • 리누스 토발즈가 리눅스 커널 소스 오픈
      • Linux OS 의 시작!
    • GNU 프로젝트 산출물과 리눅스 커널이 통합 개발

    여담 : 리차드 스톨만 (GNU 프로젝트 창시자) GNU/Linux 라고 부르기를 희망했으나 그냥 리눅스라 불림

    정리

    클라우드 컴퓨팅(AWS) 기반 리눅스 설치

    클라우드 컴퓨팅에 대해 가볍게 이해해보자!

    • 클라우드 컴퓨팅 서비스 : 서버 환경을 미리 구축해놓고, 간단한 설정으로 바로 사용할 수 있도록 만든 서비스

    • 이전 : 미리 서버 컴퓨터를 구축

      • 얼마나 쓸지 모르니까 미리 산출해서 수십, 수백억을 들여서 구축했음
      • 로그인시 등록하는 개인정보가 특정국가 밖으로 나가면 안되는 경우가 있기 때문에, 한국 서버로 올 수 없는 경우가 있었음
      • 구축 시간에만 6개월이 걸리기도함

    • 현재 : 클라우드 컴퓨팅 환경 사용

      • 서버를 구축하는데 시간이 안들음 (당장 인터넷 서비스를 열 수 있고, 바로 서버 대수를 늘릴 수 있음)
      • 초기 위험비용이 굉장히 적게듦

        • 영원히 유니크한 IP 주소 수를 주기엔 아이피 수가 부족하므로 동적으로 할당함

    ext 란?

    • Extended File System(or ext)는 Linux kernel에 구현되어 있는 파일 시스템
    • Remy Card 가 만듦(1992년)
    • 왜 만들었을까?
      • 초창기에 등장한 MINIX file system(이하 MINIX)의 한계를 극복하기 위함

    MINIX 란?

    • MINIX file system
    • 교육용으로 만 들어진 Unix-like한 운영체제 (오픈소스)

    MINIX 구조

    6 가지

    • Boot sector : 하드디스크의 첫번째 섹터

      • 컴퓨터의 구동(booting)에 필요한 기본적인 정보가 담겨있음

    • superblock : 각 파티션의 첫번째 블록

      • 파일 시스템 구조에 대한 metadata를 포함
      • metadata 를 실제 파티션으로 구분된 영역과 물리적인 디스크를 연결

    • inode : index-node의 줄임말이며, 256B 크기로 실제 파일의 저장 정보를 담고 있는 자료구조

    • inode bitmap block : 현재 어떤 inode가 사용되고 있는지 알 수 있는 블록

    • data zone : 실제 데이터가 저장되어 있는 영역

    • zone bitmap : 현재 어떤 data가 저장되어있는지 알 수 있는 bitmap

    잠깐 용어정리
    • 섹터(sector) : 하드디스크를 구역별로 나눠서 부르는 단위
    • 파티션 : 리눅스 운영체제에서 논리적으로 하드디스크를 구분하는 단위
    • 블록 : 파티션내에서 파일을 관리하는 논리적 단위

    MINIX 구조의 한계

    • 파일 이름 14글자만 가능
    • 64MB의 저장공간만 지원

    ext filesystem의 구조

    • MINIX의 구조는 계승함

      • 따라서, 구조적 한계를 극복하기 위해 기본적인 성능 향상이 필요

    • ext는 이를 극복하기 위해서 Virtual File System(VFS)를 사용!

      (source: https://opensource.com/life/16/10/introduction-linux-filesystems)

    VFS

    실제로 사용하는 파일 시스템에 관계없이 Linux Kernel이 하드웨어를 컨트롤할 수 있게끔 지원하는 구조

    VFS 역할

    • system call을 통해서 실제 파일 시스템들에게 명령을 내림
    • 이때 device driver가 특정 파일 시스템과 VFS의 system call을 소통할 수 있게 도움
      • 우리가 컴퓨터에서 특정 기기를 사용할 때, device driver를 설치해야 하는 이유
    • VFS를 통해 Linux는 파일이름을 255자까지 사용가능 지원
      • 최대 2GB의 저장 공간을 다룰 수 있게 됨

    시스템 콜을 구현한 어셈블리 코드

    실제 리눅스 운영체제에서 시스템 콜을 구현한 어셈블리 코드를 해석해보자!

    어떤 프로그램 명령을 수행한 코드일까?

    • eax, ebx, ecx, edx 레지스터의 역할
    • 0x04, 0x01이 의미
    • int 0x80의 의미

    mov eax, 1
    mov ebx, 0
    int 0x80 // 소프트웨어 인터럽트 명령
    <강의자료 발췌>
    위의 예제는 eax 레지스터에 시스템 콜 번호(1번)을 넣고, ebx 레지스터에는 해당 시스템 콜의 인자를 넣고 int 0x80인 인터럽트 를 수행하면서 시스템 콜이 수행되는 어셈블리 코드이다.

    • Interrupt Descriptor Table(IDT)

      • IDT는 해당 시스템 콜에 대한 정보를 모두 가지고 있음

    • 현재 1번 시스템콜을 호출하였고, 그 때 해당하는 integer value는 0이다.

      이를 응용해서 위의 문제를 분석하면 아래와 같다.

    1. 소프트웨어 인터럽트가 발생하면서 (시스템 콜 호출을 했기 때문에) 0x80값을 넘겨준다.
    2. CPU는 사용자 모드에서 커널 모드로 바꿔준다.
    3. IDT에서 0x80에 해당하는 주소를 찾아서 실행한다. 이 때 eax, ebx 값을 함께 넘겨준다.
    4. 불러온 시스템 콜 함수의 eax 레지스터를 확인하여 시스템 콜 번호를 IDT에서 확인한다. 현재 0x04이므로 4번 시스템 콜인 sys_write를 호출한다.
    5. sys_write 함수의 인자를 ebx 레지스터에서 확인한다. 0x01을 sys_write 함수의 인자로 넘겨준다. 6. sys_write 함수가 디스크에 write할 내용을 ecx 레지스터를 통해 확인한다. ecx 레지스터는 buffer를 저장하고 있다. buffer는 시스템 콜을 호출한 C언어로 구현된 프로그램 내에서 저장할 내용을 함께 부여받는다. (e.g. “Hello World\n”) 7. buffer의 내용중에 앞에서 부터 edx 레지스터에 할당된 크기만큼 write를 수행한다. edx에 저장된 값은 0x14이므로, 14bytes 만큼 buffer의 내용을 디스크에 저장한다.

      • gcc 명령은 주어진 C 파일을 컴파일 하는 명령어임
      • 해당 쉘에서는 키보드를 입력하면, 해당 키보드 문자가 화면에 표시하며, 엔터를 누르면 그동안 입력받은 문자열을 명령으로 인식하고, 해당 명령을 실행함
      • 스케쥴링 방식은 선점형을 지원하며, 기본적으로 멀티 태스킹을 지원함
      • 인터럽트가 발생하며 내부에서는 gcc 컴파일러가 수행됨
      • gcc 컴파일러는 주어진 옵션으로 test.c에 해당하는 test.o를 생성함
      • test.o 파일을 기준으로 실행파일인 test 파일을 생성함

    gcc 란?

    gcc(GNU Compiler Collection) : 컴파일러 드라이버 (gcc는 컴파일러가 아님)

    • 컴파일러 드라이버의 역할 : 컴파일에 필요한 여러 프로그램들을 순차적으로 호출 (실제 컴파일을 해주진 않음)

    gcc를 이용한 컴파일 과정

    gcc 는 총 4번의 호출을 함 : 전처리기 호출 -> C컴파일러 호출 -> 어셈블러호출 -> 링커 호출

    1. 전처리기 cpp0 호출
    • 전처리 단계에서 #include에 포함된 라이브러리(stdio.h)를 불러와서 현재 컴파일 대상인 코드인 test.c에 붙여넣음
    • 결과 : test.i가 생성됨
    1. C컴파일러 cc1 호출
    • cc1은 symbol table을 통하여 해당 C언어를 tokenizing -> lexical analysis -> syntax analysis -> semantic analysis를 거치고, 중간 표현 코드인 GIMPLE Tree를 생성함
      • symbol table : GNU C Compile 문법에 따름
    • GIMPLE Tree는 code optimizer와 code generator를 거쳐 최종 어셈블리 파일인 test.s로 생성됨
    1. 어셈블러 as 호출
    • as는 test.s를 test.o로 변환함 (생성함)
      • .o (object file) : ELF(Executable and Linking) 바이너리 파일 구조 규약을 따르는 형태의 바이너리 코드
    1. 링커 collect2 호출

    • 링커 : 해당 프로그램이 사용하고 있는 라이브러리 정보들(종속성을 가짐)과 해당 코드를 연결

      • 현재 사용되는 stdio.h의 내용이 이미 메모리에 있다면 따로 추가 연결을 해줄 필요가 없음
      • 만약 메모리에 해당 내용이 없다면 stdio.h을 미리 컴파일 해둔 오브젝트 코드를 가져옴
      • stdio.h : 정적 연결 라이브러리(Static Linking Library)
        • 따라서, 미리 컴파일을 해두고 .lib 파일 형태로 미리 가짐

    • 링킹을 하는 이유 : 어셈블러가 생성한 바이너리 코드파일 test.o 에는 stdio.h에서 불러와서 사용하는 코드(printf문 등)에 대한 내용이 없기 때음

      • 따라서, 실제 실행 파일을 생성할 때는 사용한 함수(printf 함수 등)에 대한 오브젝트 코드를 연결해야 실행 파일에서 문제없이 코드 순서대로 실행할 수 있음

    • 링커 호출 결과 : a.out을 생성 (최종 실행 파일)

    • gcc가 test.c를 a.out (기본 실행파일 이름)으로 생성하게 됨

    • 이 때 -o 옵션과 함께 문자열을 부여하면, 해당 문자열의 이름을 가지는 실행파일을 생성함

    • 즉, gcc -o test test.c 명령어는 test.c를 위의 과정을 모두 거쳐 최종 test라는 이름의 실행 파일을 생성해달라는 의미# API

    fork()와 exec() 시스템콜

    • fork() 시스템콜

      • 새로운 프로세스 공간을 별도로 만들고, fork() 시스템콜을 호출한 프로세스(부모 프로세스) 공간을 모두 복사

        • 별도의 프로세스 공간을 만들고, 부모 프로세스 공간의 데이터를 그대로 복사

    • exec() 시스템콜

      • exec() 시스템콜을 호출한 현재 프로세스 공간의 TEXT, DATA, BSS 영역을 새로운 프로세스의 이미지로 덮어씌움

        • 별도의 프로세스 공간을 만들지 않음

    fork() 시스템콜

    • 헤더 파일: <unistd.h>

    • 함수 원형: pid_t fork( void );

      vi forktest.c

      #include <sys/types.h>
#include <unistd.h>
#include <stdio.h>
int main ()
{
    pid_t pid;
    printf("Before fork() call\n");
    pid = fork();
    if (pid == 0)
        printf("This is Child process. PID is %d\n", pid);
    else if (pid > 0)
        printf("This is Parent process. PID is %d\n", pid);
    else
        printf("fork() is failed\n");
    return 0 ;
}

      gcc forktest.c -o forktest

      ./forktest

    • pid = fork()가 실행되면 부모 프로세스와 동일한 자식 프로세스가 별도 메모리 공간에 생성

    • 자식 프로세스는 pid가 0 으로 리턴, 부모 프로세스는 실제 pid 리턴

    • 두 프로세스의 변수 및 PC(Program Count) 값은 동일

    자식프로세스와 부모프로세스 실행 동작 다르게 가능해짐

    새로운 프로세스 공간을 별도로 만들고, fork() 시스템콜을 호출한 프로세스(부모 프로세스) 공간을 모두 복사한 후, fork() 시스템콜 이후 코드부터 실행

    execl() 시스템콜 사용예

    • execl() 시스템콜을 실행시킨 프로세스 공간에 새로운 프로세스 이미지를 덮어씌우고, 새로운 프로세스를 실행

    • perror() 함수가 호출된다는 의미는 새로운 프로세스 이미지로 덮어씌우는 작업이 실행되지 못했다는 의미

      • 즉, execl() 시스템콜 실행 실패

    • vi execltest.c

      #include <unistd.h>
#include <stdio.h>
#include <stdlib.h>
int main ()
{
    printf("execute ls\n");
    execl("/bin/ls", "ls", "-l", NULL); // 전체파일이름, argv[0], argv[1], NULL
    perror("execl is failed\n"); // 정상적인 실행에서 실행안됨
    exit(1);                     // 정상적인 실행에서 실행안됨
}

    // 환경 변수를 지정하고자 할때

    exec() 시스템콜 family 정리

    • p : path 값보고 실행하니까 전체디렉토리 필요없음
    • e : 환경변수를 별도로 정의필요

    execve() 시스템콜 예

    • 환경변수를 새로 생성하므로, 기존 사용자가 설정한 환경 변수값은 사용할 수 없음

    • 다음 코드에서, envp에는 PATH를 설정하지 않았으므로, execve()에서 “/bin/ls” 로 전체 경로를 써야, 실행 가능

    • vi execvetest.c

      #include <unistd.h>
#include <stdio.h>
#include <stdlib.h>
int main ()
{
    char *envp[] = {"USER=JEN", NULL}; // PATH 설정 안했기 때문에 오류생김
    char *arg[] = {"ls", "-al", NULL}; // /bin/ls 으로 변경해줘야 함
    printf("execute ls\n");
    execve("/bin/ls", arg, envp);
    perror("execl is failed\n");
    exit(1);
}

    • ./execvetest

    여기까지 배운 리눅스 프로세스 실행

    1. fork : 부모 프로세스로부터 새로운 프로세스 공간을 만들고 부모 프로세스 데이터 복사
    2. exec : 새로운 프로세스를 위한 바이너리를 새로운 프로세스 공간에 덮어씌움

    wait() 시스템콜

    • wait() 함수를 사용하면, fork() 함수 호출시, 자식 프로세스가 종료할 때까지, 부모 프로세스가 기다림
    • 목적
      • 자식 프로세스와 부모 프로세스의 동기화
      • 부모 프로세스가 자식 프로세스보다 먼저 죽는 경우를 막기 위해 사용
        • 위와 같은 상황에서 자식프로세스를 고아 프로세스 또는 좀비프로세스라 함

    fork(), execl(), wait() 시스템콜

    • 실제 사용 코드

      int main () {
    int pid;
    int child_pid;
    int status;
    pid = fork();                       // 새로운 프로세스공간 (based on 부모 프로세스)
    switch (pid) {
        case -1:                        
            perror("fork is failed\n");
            break ;
        case 0 :                        // 자식프로세스
            execl("/bin/ls", "ls", "-al", NULL); // ls 바이너리로 덮어씌움
            perror("execl is failed\n");
            break ;
        default :                       // 부모프로세스
            child_pid = wait(NULL);     // 기다림 (여기선 파일리스트 명령 끝날때까지)
            printf("ls is complete\n");
            printf("Parent PID (%d), Child PID (%d)\n", getpid(), child_pid);
            exit( 0 );
}

    • 왜 이런식으로 쓸까?

      • execl() 만 사용하면, 부모 프로세스가 사라짐
      • 이를 유지하기 위해, fork()로 새로운 프로세스 공간 복사 후, execl() 사용
      • wait() 함수를 사용해서 부모 프로세스가 자식 프로세스가 끝날 때까지 기다릴 수 있음

        쉘 프로그램은 fork(), exec() 계열, wait() 함수를 기반으로 작성 가능!

      #include <unistd.h>
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <sys/wait.h>
#include <sys/types.h>
#define MAXLINE 64

int main(int argc, char **argv) {
  char buf[MAXLINE];
  int proc_status;
  pid_t pid;
  printf("DaveShell ver 1.0\n");
  while(1)
  {
    memset(buf, 0x00, MAXLINE);           // buf 값 초기화
    fgets(buf, MAXLINE - 1, stdin);       // 표준입출력(standard input stream)
                                          // 키보드입력 엔터하면 stdin 으로 들어가는데, 이 때, 63bytes를 가져오겠다
    // char *fgets (char *string, int n, FILE *stream) 
    if(strncmp(buf, "exit\n", 5) == 0){   // 앞에 5 character가 같으면 종료
      break;
    }
    buf[strlen(buf) - 1] = 0x00;          // 키보드 입력후 들어온 엔터를 없애주기 위함

    pid = fork();
    if(pid==0){                           // 자식프로세스에서 buf 실행시킴 (즉, 실행파일 실행)
      if(execl(buf, buf, NULL) == -1){    
        printf("command execution is failed\n")
        exit(0)
      }
    }
    if(pid > 0) {                         // 부모프로세스에선 기다림
      wait(NULL);
    }
  }
  return 0;
}

    리눅스 프로세스 생성 속도 향상

    리눅스 프로세스 구조와 가상메모리 시스템 간의 매커니즘 이해필요

    리눅스 프로세스 구조 상세설명

    • 프로세스당 4 GB 가상 메모리 할당

    • 사용자 (0GB ~ 3 GB), 커널 (3GB ~ 4 GB)

      • 커널 스페이스에 커널이미지가 들어감

    • 커널은 동일한 이미지인데 각각의 프로세스마다 들어간다는 것은 굉장한 낭비가 아닐까?

      • 그러나 실제로 커널에 할당한 가상 메모리는 프로세스간에 공유함
        • 가상메모리에서 페이징시스템 간단 복습
          • 각 프로세스마다 페이징테이블이 생김
            • 가상메모리와 물리메모리의 주소를 매핑시켜놓은 테이블
          • 프로세스 실행중 필요에의해 커널 일부 페이지가 물리 메모리에 올라옴
          • 이 때, 동일한 페이지가 또 요청되면 테이블안에서 물리메모리 주소를 지금 들어가있는 커널 이미지의 주소로 매핑시킴
            • 즉, 별도로 올리지 않고, 동일한 주소를 페이징 테이블에 업데이트함
